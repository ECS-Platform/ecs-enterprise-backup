# ECS Benchmark — Result Interpretation

How to read each benchmark report format: what each section means, where numbers
come from, and what action to take.

> Methodology: [`BENCHMARK_METHODOLOGY.md`](BENCHMARK_METHODOLOGY.md).
> Executive justification: [`BENCHMARK_EXECUTIVE_REPORT_EXPLANATION.md`](BENCHMARK_EXECUTIVE_REPORT_EXPLANATION.md).

## Provenance banner (all reports)

Every report begins with a provenance note:
- **ESTIMATE** — documented constants × profile; not a measurement.
- Calibrate before spend decisions (see [`CALIBRATION_GUIDE.md`](CALIBRATION_GUIDE.md)).

## Executive reports

### `executive.md`
| Section | Read as | Action |
|---------|---------|--------|
| Recommended sizing table | Headline infra per profile | Compare profiles; pick target scale |
| Phase-wise sizing | Demo → Phase 1 → 100 → 500 → 2000 apps | Board/ARB rollout conversation |
| Top 5 bottlenecks | First capacity bindings at largest scale | Prioritize scaling investments |
| Top 5 risks | Operational/deployment risks | Add to risk register |
| Top 5 cost optimizations | FinOps opportunities | Evaluate lifecycle, compression, CUD |

### `executive.html`
Same content as `executive.md` in a single-file HTML dashboard. Open in a browser
for CIO/CTO review. No dependencies.

### `executive.csv`
One row per profile; columns: `profile`, `apps`, `replicas`, `gke_nodes`,
`cloud_sql_tier`, `gcs_year1_gib`, `gcs_year5_gib`, `cost_monthly`, `cost_5yr`.
Import into spreadsheets for comparison charts.

### `executive.json`
Structured `build()` output: `sizing_table`, `phase_wise_sizing`, `top_bottlenecks`,
`top_risks`, `top_cost_optimizations`, `_disclaimer`. Use for programmatic
dashboards or CI baseline comparison.

## Section reports (`capacity_{section}.md`)

Generated by `report.SECTION_REPORTS` when running `--all` (or individually via
`--cpu`, `--ram`, etc.).

### CPU report (`capacity_cpu.md`)
- **Peak cores** — busiest-minute CPU demand (coarse + detailed breakdown).
- **core-hours/day table** — top operations by CPU cost; use to find optimization targets.
- **Not measured** — per-op CPU-ms are constants; calibrate from telemetry.

### RAM report (`capacity_ram.md`)
- **Peak total GiB** — sum of per-consumer peaks.
- **High-water mark MiB** — worst-case concurrent consumers.
- **Per-consumer table** — which workload classes dominate RAM (prompt, connector, cache).
- Compare peak/replica vs pod memory request in `kubernetes` section.

### Storage report (`capacity_storage.md`)
- **Year 1 / retention GiB** — raw object storage before and after lifecycle.
- **Dedup + compression** — efficiency savings (not yet applied to cost by default).
- **Retention projection** — 1/3/5/7/10 year cumulative GiB.
- Use for GCS bucket planning and lifecycle policy design.

### Database report (`capacity_database.md`)
- **Cloud SQL tier** — starting machine type from year-1 GiB band.
- **Year 1 / year N GiB** — metadata + vectors + growth.
- **WAL / backup / restore** — operational sizing for DBA review.
- **Partitioning strategy** — when tables should be partitioned.

### Network report (`capacity_network.md`)
- **Connector ingress GiB/day** — per-connector payload aggregation.
- **Cross-cloud AWS↔GCP** — egress for Net Banking ↔ GCP Mobile Banking.
- **Evidence upload/download** — object transfer volume.
- **Total egress/month** — feeds cost `network_egress` line item.

### Cost report (`capacity_cost.md`)
- **Monthly / annual / 5-year totals** — illustrative TCO.
- **Component breakdown** — compute, SQL, storage, logging, monitoring, network, LB.
- **Disclaimer** — rates are illustrative; replace `CostRates` before budget approval.

## Main capacity reports

### `capacity.json`
Full estimate dict per profile: all 18 sections (`profile`, `gke_compute`,
`node_pool`, `postgres_pgvector`, `gcs_object_storage`, `logging_monitoring`,
`cpu_breakdown`, `ram_breakdown`, `db_durability`, `object_storage_detail`,
`network`, `cost`, `kubernetes`, etc.) plus `recommendation_table`.

### `capacity.md`
Human-readable report with recommendation summary table + per-profile detail
(GKE, PostgreSQL, GCS, logging sections).

### `capacity.csv`
Flat recommendation row per profile (same columns as executive CSV plus extra
fields like `peak_cores`, `logs_per_day_gib`).

## Advanced reports (stdout or optional files)

### Stress report (`--stress` / `--stress-scenario`)
Printed to stdout as JSON. Each scenario has:
- `impact_factor` — multiplier on baseline activity.
- `bottleneck` — predicted binding resource.
- `mitigation` — recommended response.
Interpret as **modeled worst-case**, not measured limits. Validate with load testing.

### Telemetry report (`--telemetry`)
Printed to stdout as JSON from `RuntimeTelemetry`. Fields: CPU %, RSS, disk I/O,
network, elapsed time. This is **measured** (of the benchmark run itself), not
workload capacity. Use as calibration input.

### Calibration report (`--calibrate observed.json`)
Printed to stdout. Per metric:
- `estimated` vs `observed` vs `calibration_factor` (observed ÷ estimated).
- `recommended_new_constant` — suggested override (apply explicitly).
- `confidence` — low until multiple calibration cycles.

## Estimated vs measured vs calibrated

| Report type | Default mode | Becomes measured when | Becomes calibrated when |
|-------------|--------------|----------------------|------------------------|
| Capacity / executive | Estimated | — | Constants overridden after `calibrate()` |
| CPU / RAM sections | Estimated | `RuntimeTelemetry` + load test | `WorkloadConstants` updated |
| Token feed | Estimated (chars/4) | `measured_tokens` or audit-LLM benchmark | `measured_tokens=` passed in |
| Cost | Estimated (illustrative rates) | GCP billing export | `CostRates` replaced |
| Telemetry | **Measured** | Always (of benchmark process) | N/A |
| Calibration | Compares est vs obs | Requires observed JSON input | Output drives constant updates |
| Stress | **Modeled** | Requires executed load test | Multipliers tuned manually |
