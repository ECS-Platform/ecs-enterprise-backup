# ECS Predefined Queries by Technology

## Purpose

A complete, technology-grouped inventory of every **predefined query (control)**
available in this ECS (Evidence Collection System) instance. It lets an auditor,
developer, or operator see — per technology — which read-only checks exist, what
they collect, their framework mapping, how they execute, and where they are
defined. This is a **documentation-only** inventory generated by reading the live
catalog through the existing predefined-query engine; it defines **no new
queries** and changes no runtime behaviour.

> **Verified totals (this instance):** **187 controls** across **21 technologies**.
> Naming reflects the catalog's canonical labels (e.g. `Aurora MySQL`,
> `YugabyteDB`, `Red Hat Enterprise Linux 8.x/9.x`). `Unknown` is a bucket of
> uncategorised Excel-sourced controls (no connector). Counts, names, and mappings
> below were **extracted from code — not assumed**.

## Repository source references

| Source | Path | Role |
|--------|------|------|
| Predefined query engine | `modules/operations/engines/predefined_queries_engine.py` | Loads controls, technology detection (`TECHNOLOGY_RULES`), dispatch (`run_predefined_query`), filter options |
| Supplementary catalog | `modules/operations/engines/supplementary_query_catalog.py` | Code-defined additive queries (all `*X-*` / `ASX-*` IDs) |
| Excel control library | `ECS_Query_Driven_Control_Library_Consolidated.xlsx` (repo root) | Primary workbook controls (`DB-*`, `OS-*`, `PCI-*`, `MW-*`, `APP-*`, `AI-*`, …) |
| Connector registry | `modules/operations/engines/query_connectors.py` (`connector_for_technology`) | Maps technology → connector |
| Query connectors | `modules/operations/engines/*_connector.py` | Per-technology execution (SQL / shell / command-spec) |
| Technology fingerprinting | `modules/audit_intelligence/engines/technology_fingerprint.py` | Asset → technology classification |
| Asset-driven scheduler | `modules/audit_intelligence/services/asset_scheduler.py` | Classifies assets, routes to baseline collector |
| Technology→control mapping | `modules/audit_intelligence/engines/technology_control_mapping.py` | Read-only projection over the catalog |

**Execution type legend:** `SQL` (DB driver) · `shell/CLI` (docker-exec / CLI, e.g.
`asinfo`, `redis-cli`, `kubectl`) · `command-spec` (MongoDB admin command name) ·
`SSH/remote` (Windows via SSH) · `other` (Excel-sourced command not gated to a live
connector). **Runnable** = control is live-execution-enabled (Run Query works).

## How these queries are used

- **Predefined query engine** (`predefined_queries_engine.py`) — loads all controls
  (Excel + supplementary), assesses execution capability, exposes
  `get_all_controls()` / `get_technology_filter_options()` / `filter_controls()`
  and dispatches live execution via `run_predefined_query(control_id, user)` →
  a per-technology runner → connector. The Run Query UI
  (`/mvp/predefined-queries` + `/run`) is server-rendered from this data.
- **Technology fingerprinting** (`technology_fingerprint.py`) — classifies an asset
  to one of these technologies by image/name/port/hint, so the applicable controls
  can be selected automatically.
- **Technology rules** (`TECHNOLOGY_RULES` in the engine) — regex rules that map an
  Excel control's `query` text to a technology when no explicit technology is set;
  supplementary entries carry an explicit `technology` field instead.
- **Connector routing** (`connector_for_technology` in `query_connectors.py`) —
  maps each technology to the connector used for live execution and for the UI's
  Run Query readiness check (`prepare_execution`).
- **Evidence planner / asset-driven scheduler** (`asset_scheduler.py`) — for each
  classified asset, routes to the *baseline collector* (this predefined-query
  engine) with the technology's control IDs, or to an enterprise connector; the
  dry-run planner reports what *would* run without executing anything.
- **Technology→control mapping** (`technology_control_mapping.py`) — a read-only
  projection over this same catalog used by Audit Intelligence (dashboards,
  framework readiness) — it never mutates the catalog.

## Gaps

- **Technologies with no predefined queries.** The demo Docker source connectors
  **Jenkins** and **Gitea** exist as `connectors:` targets but have **no predefined
  queries** (they are not query technologies). Enterprise integration adapters
  (ServiceNow, Archer, SharePoint, Teams, Outlook, Jira, Confluence, SonarQube
  *adapter*, Checkmarx, Prisma Cloud, Tripwire) collect evidence via the
  integrations layer, **not** via predefined queries — expected by design.
- **Queries without control/framework mapping.** **None** — all 187 controls carry
  a `framework_coverage` value.
- **Queries without connector mapping.** **15** controls in the `Unknown` bucket
  (`DB-005`, `MW-002`, `PCI-002`, `PCI-004`, `APP-003`, `DPSC-001/002`,
  `VAPT-001/002/003`, `ITPP-001/004`, `AI-001/002/003`) have no technology → no
  connector → not live-executable. They remain catalogued for coverage.
- **Not runnable in the demo (mapped but gated).** Oracle (2), Aurora MySQL (1),
  YugabyteDB (1), Windows (3), NGINX (1), Tomcat (1), Trivy (1) and various
  workbook OS controls report `Configuration Required` / `Dependency Missing` /
  `Connector Missing` until a live target + driver are configured.
- **Duplicate / overlapping queries.** No **duplicate control IDs** (0). There is
  intentional **functional overlap** between older workbook controls and the newer
  supplementary controls (e.g. `DB-001` `SHOW ssl;` vs `PGX-001`; `DB-006` vs
  `YBX-001`; `DB-007` vs `MYX-001`). The workbook entry always wins on ID collision
  (supplementary is additive); the overlap is by design (workbook = original,
  supplementary = richer, connector-ready variants).
