## 1. Product Vision

### 1.1 Purpose of ECS

ECS (Enterprise Compliance System) is a **single-pane banking Governance, Risk, Compliance, Audit and Evidence Management platform** designed to replace fragmented spreadsheets, ServiceNow queues, SharePoint repositories and auditor email threads with a unified, role-aware control surface.

The platform is currently positioned as an **enterprise-grade interactive demo / walkthrough application** intended to be shown to:
- CIOs and CISOs evaluating the governance product
- Auditors (RBI, Deloitte, EY, KPMG, PwC) running mock readiness reviews
- App Owners and Compliance Officers experiencing day-in-the-life workflows
- Regulators and board members reviewing executive heatmaps

It is **not yet a transactional production system** — all data is generated in memory and is deterministic (hash-seeded) so the demo state remains stable across page loads.

### 1.2 Business Objectives

The codebase encodes the following business objectives, derived from module purpose strings (`app/module_capabilities.py::MODULE_PURPOSES`) and engine docstrings:

1. **Replace static spreadsheets** with a centralized, dynamic, filterable governance surface (every page consumes the same filter engine).
2. **Audit readiness as a continuous discipline** — quarterly and annual audit calendars, readiness scoring, mock-audit packs, baselining history, and a "war-room" Audit Prep cockpit.
3. **Evidence reuse across frameworks** — map once, satisfy multiple controls; reduce duplicate uploads through control theming and cross-framework overlap matrices.
4. **Framework on-boarding intelligence** — ingest novel compliance frameworks, normalize controls into canonical themes, detect duplicates/overlaps, score reuse potential, and calculate audit effort saved.
5. **Role-tailored experiences** — App Owner, Auditor, CIO, Vertical Head, Compliance Head, Compliance Officer, Functional Head each get distinct dashboards, KPIs, queues and permissions.
6. **AI Copilot for governance** — natural-language audit queries, evidence definitions, framework explanations, action shortcuts, and an AI Governance posture surface (prompt audit, hallucination alerts, unsafe prompts, token usage).
7. **Enterprise integration story** — ServiceNow GRC/CMDB, SharePoint, Jira, Prisma, Tripwire, SonarQube, Checkmarx, SIEM, Teams (all mocked) feed evidence into a single repository.

## 6. Current Workflows

### 6.1 Evidence Lifecycle

State machine: `app/evidence_workflow_engine.py`.

```
Owner states:           Auditor states:
─────────────           ────────────────
Draft                    Pending Review
Uploaded                 Under Validation
Submitted to Auditor     Approved
Re-upload Requested      Rejected
Rejected by Auditor      Re-upload Requested
Approved by Auditor      Escalated
Clarification Required   Exception Raised
Draft Cancelled          Under Validation (clarification)
```

State transitions (from `app/main.py` + `app/routes_mvp.py`):

```
Draft ── upload ──▶ Uploaded
Uploaded ── submit ──▶ Submitted to Auditor ─▶ Auditor:
                                                 ├── approve  ──▶ Approved (observation closed)
                                                 ├── reject   ──▶ Rejected (Owner re-upload required)
                                                 ├── clarify  ──▶ Clarification Required
                                                 ├── request_reupload ──▶ Re-upload Requested
                                                 └── escalate ──▶ Escalated to leadership
Rejected ──▶ Owner team_resubmission ──▶ revised_upload ──▶ reevaluate ──▶ ready_resubmit ──▶ submit
Approved ──▶ Observation Closed (terminal)
Cancelled (terminal)
```

Persistence: `ecs_state.submitted_controls`, `approved_controls`, `rejected_controls`, `escalated_controls`, `clarification_controls`, `cancelled_drafts`, `owner_drafts`, `owner_comments`, `submitted_meta`, `closed_observations`.

Resubmission gating: `app/resubmission.py` enforces `can_resubmit_to_auditor(key)` so owners cannot submit until they have walked through the resubmission stages.

### 6.2 Audit Lifecycle

Orchestrated by `app/audit_schedule_engine.py`.

```
quarter_window calc (TODAY = 2026-05-28)
  │
  ▼
generate_upcoming_audits() ──┐
generate_baselining_history()│
generate_audit_calendar()    │──▶  build_audit_operations() ──▶ /mvp/audit-prep
generate_preparation_pipeline()│
build_kpi_drilldowns()       ──┘

Stages used on Upcoming Audits table:
  Scheduled · Evidence Collection · Auditor Review ·
  Observation Raised · Revalidation · Closure Pending · Closed
```

For each audit record: framework, application, auditor, scope hint, audit type (quarterly/annual), start/end dates, evidence submission deadline, readiness %, missing controls, pending evidence, blockers, owners, escalation route. Stage transitions are projected from the in-memory workflow state.

### 6.3 Compliance Lifecycle

`app/governance_lifecycle_engine.py` defines an end-to-end compliance lifecycle that the Lifecycle page renders:

```
CONTROL_STAGES:
  Mapped → Implemented → Validation Pending → Failed → Exception Approved →
  Remediation Open → Revalidated → Approved → Archived

EVIDENCE_STAGES:
  Requested → Uploaded → Validation Pending → Approved →
  Stale → Refreshed → Reused → Archived

OBSERVATION_STAGES:
  Open → Assigned → Under Review → Remediation In Progress →
  Pending Validation → Closed → Reopened

REMEDIATION_STAGES:
  Planned → In Progress → Awaiting Validation → Delayed → Escalated → Completed

AUDIT_STAGES:
  Audit Scheduled → Evidence Collection → Auditor Review →
  Observation Raised → Revalidation → Closure Pending → Closed

EXCEPTION_STATUSES:
  Active · Expiring Soon · Revalidation Required · Expired · Closed
```

### 6.4 Governance Lifecycle (Exceptions / TD / Risk)

`app/enterprise_grc.py` + `app/exception_state_engine.py`:

```
Raise Exception ──▶ Active ──▶ Expiring Soon ──▶ Revalidation Required ──▶
                                                  │
                                                  └─▶ Expired ──▶ Closed
Risk Register treatments: Mitigate · Accept · Transfer · Avoid
TD types: Risk Acceptance · Compensating Control · Temporary Waiver · TD Extension
```

Audit trail: every governance action emits `audit_trail.log_event(...)` with timestamp, actor, role, framework, control, detail, evidence_id; surfaces in notifications and recent activity feeds (200-event ring buffer, 30-notification ring buffer).

### 6.5 Application Onboarding Lifecycle

`app/onboarding_engine.py` + `app/templates/mvp_onboarding.html`:

```
1. App metadata captured (name, owner, vertical, criticality, tech stack)
2. Applicable frameworks auto-suggested (FRAMEWORK_META scoring)
3. Controls auto-mapped per framework (deterministic per-app seeding)
4. Initial readiness % calculated
5. Onboarding modals → register, set scope, generate evidence requests
6. Application appears in:
     - ecs_state.onboarded_applications
     - all per-framework dashboards
     - audit schedules
     - reuse / heatmap calculations
```

Framework onboarding (separate flow for new compliance frameworks): `app/framework_onboarding_engine.py` walks:

```
LIFECYCLE_STATES = (Draft → Imported → Mapped → Reviewed → Approved → Active)
```

Plus the new Framework Loader (`app/framework_loader_service.py`) which adds:
- CSV/Excel parsing with animated log
- Control extraction → ECS Control ID generation
- Reusable control detection (theme overlap)
- Evidence reuse calculation
- Audit effort estimate
- Activation

### 7.2 Relationships

```
Framework 1───* Control 1───* Evidence
Control  1───* Observation 1───* Remediation
Application 1───* Evidence (via application_name)
Application 1───* Server (CMDB)
Application *───* Framework (via applicability)
Theme    *───* Control      (via framework_intelligence.classify_control_themes)
Evidence *───* Framework    (reuse via evidence lineage / overlap matrix)
Risk     *───1 Control      (linked_control)
Exception *───1 Control     (compensating)
Audit    *───* Framework   (1:1 per cycle) + *───1 Application
Connector *───* Evidence    (imported_evidence, mapped_controls)
Prompt   *───1 Application  (via team / app attribution)
Hallucination 1───1 Prompt
Token Usage   *───1 Team, *───1 Application
```

### 7.3 Lineage concepts

`generate_evidence_lineage()` produces an explicit evidence lineage graph:

```
EVD-<base>  ──spawn──▶  EVD-<derivative-1> (framework B, app A)
                     ──▶  EVD-<derivative-2> (framework C, app A')
                     ──▶  EVD-<derivative-3> (framework B, app A'')
```

Each derivative records `original_framework`, `original_app`, `reuse_count`. This is rendered on the Demo Overview page as a "lineage graph" table.

Framework reuse traceability (`framework_intelligence.build_reuse_traceability`) gives:
```
Theme ──▶ Frameworks ──▶ Applications ──▶ Reuse targets
```

## 9. Technical Architecture

### 9.1 Frontend structure

- **Stack:** Jinja2 server-rendered HTML + Bootstrap 5.3 + handcrafted CSS in `enterprise_theme.html` and per-page inline `<style>` blocks. No SPA, no React/Vue.
- **Layout:** 2-column (sidebar + main content); main content uses zone-based macros (`zone_open`, `zone_close`, `kpi_card`, `accordion_section`, `page_header`) from `partials/ecs_ux_macros.html`.
- **Templates:** 39 page templates in `app/templates/`, 62 partials in `app/templates/partials/` (15,915 total lines).
- **Interactivity:** Bootstrap collapses + small inline JS for modals, KPI drill-downs, and filter clients. JSON APIs at `/api/…` power click-through detail modals (e.g. `ecsAuditKpiModal`, `ecsAuditDetailModal`, `ecsDemoDrillModal`).
- **State propagation:** All page state lives in URL query params (`?role=…&user=…&framework=…&fw_app=…`). No client-side state library.
- **Theming:** Shared classes (`.ecs-page`, `.ecs-page-header`, `.ecs-kpi-card`, `.ecs-fw-pci`, `.ecs-priority-grid`); per-framework accent colors in `framework_dashboards.FRAMEWORK_THEMES`.
- **Filter engine UI:** `partials/standard_filter_client.html`, `analytics_filter_client.html`, `lifecycle_filter_client.html`, `comparison_filter_client.html`, `completeness_filter_client.html`, `operations_filter_client.html`, `executive_dashboard_client.html`, `gap_export_client.html`, `integrations_hub_executive_client.html`.
- **Critical shared macros:** `ecs_ux_macros.html` (page_header, kpi_card, accordion_section, priority_card, zone_open/close, role_label, safe_get, governance_action_menu, evidence_action_menu, empty_state). These macros are *strict-undefined-safe* after the Jinja Template Recovery phase.
- **AI Copilot UI:** `partials/chatbot_global.html` — floating chatbot dock visible across all pages.

### 9.2 Backend structure

- **Stack:** FastAPI (latest) + Uvicorn + Jinja2Templates + python-multipart. No SQL, ORM, or async DB driver.
- **Process model:** Single-process (`uvicorn app.main:app --reload`). Lifespan handler in `app/main.py::ecs_lifespan` seeds demo state on boot.
- **Routing:**
  - Core auth + dashboards + framework + evidence-review workflows in `app/main.py` (~1,100 lines, 35+ endpoints).
  - MVP modules + APIs registered via `register_mvp_routes(app, templates)` in `app/routes_mvp.py` (~1,600 lines, 70+ endpoints).
  - Evidence-specific REST in `app/evidence_routes.py`.
- **Module structure (selected key files):**
  | Concern | Module |
  | --- | --- |
  | Shared state | `ecs_state.py` |
  | Catalog | `framework_catalog.py` (15 framework catalogs) |
  | Workflow engine | `evidence_workflow_engine.py`, `workflow_module.py`, `resubmission.py` |
  | Audit prep engine | `audit_schedule_engine.py` |
  | Mock data engine | `ecs_mock_engine.py` |
  | Framework intelligence | `framework_intelligence.py` |
  | Framework onboarding | `framework_onboarding_engine.py`, `framework_loader_service.py` |
  | Evidence repository | `evidence_repository.py`, `evidence_health_engine.py`, `evidence_approval_engine.py` |
  | Governance | `governance_lifecycle_engine.py`, `governance_completeness_engine.py`, `governance_intelligence.py`, `governance_data_enrichment.py` |
  | Chatbot | `chatbot_engine.py`, `chatbot_enhanced.py`, `chatbot_context_engine.py`, `chatbot_nav.py` |
  | Enterprise GRC | `enterprise_grc.py`, `enterprise_context.py`, `enterprise_mock_service.py`, `executive_analytics_engine.py` |
  | Integrations | `integrations_module.py`, `integration_health_engine.py`, `integration_hub_executive_engine.py` |

- **Configuration:** No `.env`, no settings module. All configuration is hard-coded constants.
- **Startup:** `start_ecs.sh` (`pkill -f uvicorn && pip install … && uvicorn app.main:app --reload`).

### 9.3 Database structure

There is **no database**. Persistence is entirely in-process Python data structures:

- Dicts / lists in `app/ecs_state.py` (workflow state, registries, ring buffers).
- Module-level constants in `framework_catalog.py`, `integrations_module.py`, `enterprise_grc.py`, etc.
- Ring buffers in `audit_trail.py` (200 events / 30 notifications).
- Deterministic regeneration on demand via `hashlib.md5`-seeded helpers in mock engines.

Implications:
- State resets on process restart, then is reseeded by `seed_demo_workflow_state()` (idempotent).
- Concurrent writes are not protected — single-user demo workload only.
- No multi-tenant separation.

### 9.4 Mock data architecture

Three layers of mock data, each with different responsibilities:

| Layer | Module | Used by |
| --- | --- | --- |
| **Catalog layer** | `framework_catalog.py` | All page renders; provides the canonical control + evidence structure. Hand-curated controls, deterministic evidence via `_make_evidence` + `_stable_pick`. |
| **Engine layer** | `audit_schedule_engine.py`, `framework_intelligence.py`, `evidence_workflow_engine.py`, `framework_onboarding_engine.py`, `governance_lifecycle_engine.py`, `comparison_engine.py`, `enterprise_grc.py`, `correlation_engine.py`, `operations_mock_data.py`, `operational_workflows.py` | Per-module dashboards. Pure functions, all hash-seeded. |
| **Demo façade** | `ecs_mock_engine.py` | Demo Overview page + `/api/demo/*` endpoints; wraps the engine layer and adds banking applications registry, ServiceNow tickets, AI governance, multi-year audit history, risk heatmap, baselining drift, evidence lineage, VAPT findings, CIO executive snapshot. |

Determinism contract: every record's identity comes from `_seed(*parts) = int(md5("::".join(parts)).hexdigest(), 16)` so the same (framework, app, control, year, …) tuple always produces the same record across reloads.

`DEMO_MODE = True` and `DEMO_ANCHOR_DATE = date(2026, 5, 28)` in `ecs_mock_engine.py` anchor the demo timeline.

## 11. Recommended Next Phase

The following sequencing balances *demo fidelity* (immediate stakeholder value) with *technical hardening* (production-readiness). Recommendations are concrete and ordered.

### 11.1 Stabilize the demo posture (1–2 weeks)

15. **Wire a real LLM** behind `chatbot_enhanced.py` (e.g. Bedrock / Azure OpenAI / Anthropic) with strict structured-output contracts (function-calling) so answers are always anchored to ECS data.
16. **Replace heuristic semantic search** in `search_module.build_search_discovery()` with vector embeddings (OpenAI `text-embedding-3-small`, pgvector or Chroma) over evidence summaries + control descriptions.
17. **Implement real evidence sufficiency scoring** — chain (a) attribute extraction (LLM), (b) control-vs-evidence match score, (c) reviewer-style explanation, surfaced in `evidence_review.html`.
18. **Wire hallucination + unsafe-prompt guards** into the real prompt pipeline (perplexity score, PII detector, prompt-injection classifier). Replace the mock alerts.
19. **Token spend governance** — instrument every LLM call with team / app attribution and integrate with the provider's billing API for real cost dashboards.

### 11.5 Enterprise integration (4–6 weeks)

20. **Replace `integrations_module._connectors`** with real connectors:
    - ServiceNow REST (Tickets, CMDB CIs)
    - SharePoint Graph (evidence library)
    - Jira (epics → exceptions)
    - Tripwire / Prisma / SonarQube / Checkmarx (scheduled pulls)
    - Splunk SIEM (alert ingestion)
21. **Add scheduler workers** (Celery + Redis or APScheduler) so `scheduler_module.run_scheduled_pull` is genuinely time-driven.
22. **Real-time push** for KPI counters (SSE or WebSocket) — particularly useful on dashboards during a live audit.

### 11.6 Production hardening (4–6 weeks)

23. **CI/CD** — GitHub Actions running `pytest`, `mypy`, template validator, container build, vulnerability scan (`pip-audit` + `trivy`).
24. **Containerize** with a multi-stage Dockerfile + docker-compose for local dev (app + db + redis + minio).
25. **Observability** — OpenTelemetry tracing on every request, structured JSON logs, Sentry for exceptions, Prometheus metrics for evidence backlog / approval SLAs.
26. **Multi-tenant scaffolding** — `tenant_id` on every entity; URL prefix per tenant.
27. **Security review** — CSRF tokens on every POST, signed URLs for downloads, content security policy, role-based authorisation tests.

### 11.7 New product features

28. **Mobile-friendly views** — auditor's "approve from phone" surface.
29. **Auditor evidence packs** — one-click export of every evidence file + approval trail for a given audit cycle (today only metadata exports).
30. **Regulator dashboard** — read-only view scoped to regulatory frameworks (PCI DSS, DPSC, RBI Cyber Security, CSITE) with anonymized application names.
31. **Vendor / Third-Party module** — currently only a control theme; promote to a full module with vendor risk scoring, SOC2 ingestion, contractual evidence vault.
32. **AI Governance framework** — add AI Governance as a first-class entry in `FRAMEWORK_CATALOG` (controls: model registry, prompt audit, hallucination thresholds, bias review, training data lineage, model retraining cadence) so it lives alongside PCI DSS / ISO27001.
33. **Continuous controls monitoring (CCM)** — schedule `control_validation_engine` checks on a cadence and visualize drift over time.
34. **Audit response narrative generator** — given an observation, generate a draft management response using the LLM, grounded in linked evidence.
